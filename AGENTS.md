# AGENTS.md

このリポジトリでは、AIエージェントを使って個人開発を進めます。
AIエージェントは、以下のルールに従って作業してください。

## 基本方針

- GitHub Issueを作業単位とする
- Codexを実装、テスト、Git操作、PR作成の主担当とする
- Gemini、Antigravityなどの補助エージェントは、別解の検討、調査、UI案の比較に使う
- 複数のAIエージェントで同じブランチを同時に編集しない
- Issueに書かれていない機能を勝手に追加しない
- 実装前にIssueの目的、実装範囲、受け入れ条件、スコープ外を確認する
- 技術スタック、提供形態、データ保存方式を勝手に変更しない
- 変更が必要な場合は、理由と影響範囲を説明してから進める
- MVPではシンプルさと公開可能性を優先する
- 過剰設計を避ける
- 今日中に動かせる粒度、公開できる粒度を優先する
- ユーザー体験に直接関係しない凝った実装を増やさない

## 作業開始前に確認すること

各Issueに着手する前に、以下を確認してください。

- このIssueの目的は何か
- 実装範囲はどこまでか
- スコープ外は何か
- 依存Issueは完了しているか
- 受け入れ条件は明確か
- 既存実装への影響はあるか
- どの検証コマンドを実行すべきか

不明点があっても、Issueと既存コードから安全に判断できる場合は、合理的な仮定を置いて作業を進めてください。
ただし、スコープや仕様が変わる判断が必要な場合は、実装前に確認してください。

## 設計Issueのルール

設計Issueでは、すぐに実装コードを書かないでください。

以下を整理してください。

- 解決したい課題
- 想定ユーザー
- MVP範囲
- 今回作らないこと
- 画面構成
- 入力項目、出力項目
- 技術スタック候補
- データ保存方式
- 公開方法
- 将来の拡張余地
- 実装Issueへの分解案

技術選定では、最初から特定の技術に固定せず、MVPの公開しやすさ、実装速度、保守性を比較してください。

## 実装Issueのルール

実装Issueでは、設計Issueや既存ドキュメントで決めた内容に従ってください。

- できるだけ小さな差分で実装する
- 1つのIssueで関係ないファイルを変更しすぎない
- 既存の構成、命名、スタイルに合わせる
- 型安全性を意識する
- 入力バリデーションを最低限入れる
- エラー時や空状態でアプリがクラッシュしないようにする
- UIはまず標準的で分かりやすいものにする
- localStorageなど端末内保存を使う場合は、読み込み失敗時にクラッシュしないようにする
- 外部API、ログイン、課金、広告SDK、解析SDKはIssueで明示されない限り追加しない

## コード変更時の品質チェック

AIエージェントがアプリケーションコード、設定、テストを変更した場合は、コミット前に変更内容に応じて以下を実行してください。

```bash
pnpm format:check
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

- 原則として、コードを変更したIssueでは上記をすべて実行する
- 実行できないコマンドや失敗したコマンドがある場合は、原因と影響を明記し、成功したものだけを報告して完了扱いにしない
- ドキュメントのみの変更では、Markdownの差分チェックを行い、アプリケーションの品質チェックは変更内容に応じて省略できる
- フォーマット修正が必要な場合は`pnpm format`を実行してから再確認する

実装後は以下を報告してください。

- 実装概要
- 変更ファイル
- 動作確認結果
- 未対応事項
- 懸念点
- 次に進めるIssue

## 実装依頼への対応フロー

ユーザーから実装を依頼された場合は、返答する前に原則として以下まで進めてください。

1. 対象Issueと関連ドキュメントを読む
2. 実装する
3. 関連する検証コマンドを実行する
4. 変更内容を確認する
5. コミットする
6. ブランチをpushする
7. GitHubのpull requestを作成する
8. PRリンクと検証結果を返答する

PRは、明示がない限りDraftではなくOpen状態で作成してください。
リポジトリの権限や認証の都合でpushやPR作成ができない場合は、その時点までの作業内容と未完了の手順を明確に報告してください。

## レビュー対応

レビューコメントに対応する場合は、以下を守ってください。

- コメントの指摘内容を確認する
- 必要最小限の差分で修正する
- 修正後に関連する検証コマンドを実行する
- 変更をコミットしてpushする
- 対象PRへ対応内容をコメントする

CodexのPRレビューはすべてのPRで使い、GitHub Actionsと人間の確認を補完してください。レビュー指摘は、対象Issueの目的、受け入れ条件、既存仕様への影響を確認してから対応します。

## PRレビュー方針

GitHub上で動作するレビューエージェントを含め、PRレビューでは次を優先して確認してください。

- 対象Issueの受け入れ条件を満たしているか
- スコープ外の実装が混入していないか
- 既存仕様を壊す変更がないか
- 入力バリデーション、空状態、エラー処理に漏れがないか
- 保存データの互換性を壊していないか
- セキュリティ、認証、外部API、課金に影響がある場合は重点的に確認する
- 確実に問題となる指摘だけをコメントし、好みや任意改善は指摘しない

PR本文と紐づくIssueをレビュー根拠にし、Issueが確認できない場合は仕様上の推測による指摘を避けてください。

## レビュー時の確認観点

レビューでは以下を確認してください。

- Issueの受け入れ条件を満たしているか
- スコープ外実装が入っていないか
- ビルドエラーがないか
- テスト、lint、型チェックが通るか
- 入力エラーや空状態に対応しているか
- 保存処理や既存データの読み込みが壊れていないか
- UIが極端に崩れていないか
- スマホで主要操作ができるか
- 公開や審査提出に支障がないか
- 過剰実装になっていないか

## 禁止事項

- Issueにない機能追加
- 技術スタックの無断変更
- 不要な外部ライブラリ追加
- 広告SDKや解析SDKの無断追加
- ログイン機能の無断追加
- DBやサーバー処理の無断追加
- 既存仕様を壊す大規模リファクタリング
- READMEや設定ファイルの不要な書き換え
- 秘密情報、APIキー、個人情報のコミット

## 判断基準

迷った場合は、以下を優先してください。

1. 今日中に動くこと
2. 公開または審査提出できること
3. スコープが小さいこと
4. ユーザーが使えること
5. 後から拡張できること

完璧な設計より、公開可能なMVPを優先してください。
